🛠 técnicas

Técnicas de Padronização Textual

✂️

Strip & Trim

Remove espaços em branco nas bordas e normaliza múltiplos espaços internos.

🔡

Case Normalization

Lowercase, uppercase ou title case consistente para garantir comparações corretas.

🔤

Remoção de Acentos

Unidecode ou unicodedata para normalizar caracteres e permitir joins seguros.

🔍

Regex Patterns

Extração e substituição estruturada: CPF, CNPJ, telefone, CEP, moeda.

📖

Dicionários de Mapeamento

De → para fixo para abreviações, siglas e sinônimos de domínio.

🎯

Fuzzy Matching

Distância de Levenshtein para agrupar valores similares (fuzzywuzzy / rapidfuzz).

Padronização Básica com Pandas

cleanse.py
import pandas as pd import re # Normalização de texto: strip, lowercase, espaços def normalizar_texto(valor: str) -> str: if pd.isna(valor): return pd.NA valor = str(valor).strip().lower() return re.sub(r'\s+', ' ', valor) # Substituir nulos disfarçados NULOS = {'n/a', 'na', '—', '-', '.', 'null', ''} df['coluna'] = df['coluna'].apply( lambda x: pd.NA if normalizar_texto(x) in NULOS else normalizar_texto(x) )
fuzzy matching

Fuzzy Matching — Similaridade Textual

Nem sempre dois textos idênticos representam a mesma entidade — e o contrário também é verdade. O fuzzy matching mede o grau de semelhança entre strings usando distâncias como Levenshtein (quantas edições mínimas transformam uma string em outra).

Entrada Candidato Score
Sao Paulo São Paulo 97
Campians Campinas 88
Rio de Jneiro Rio de Janeiro 93
Manaus Brasília 12

Score de 0 a 100. Um limiar (threshold) define o que é aceito — valores abaixo ficam como nulo para revisão manual.

fuzzy_match.py
from rapidfuzz import process, fuzz CIDADES = ["São Paulo", "Campinas", "Rio de Janeiro", "Manaus"] def corrigir_cidade(entrada, threshold=85): # token_sort_ratio: ignora ordem das palavras match, score, _ = process.extractOne( entrada, CIDADES, scorer=fuzz.token_sort_ratio ) # Abaixo do threshold → NaN (revisão manual) if score >= threshold: return match return pd.NA

encoding

Encoding — Quando os bytes mentem

Windows-1252 misturado com UTF-8

Quando linhas de um CSV são salvas por sistemas diferentes — por exemplo, uma exportação do Windows e outra do Linux — o mesmo arquivo pode conter dois encodings distintos. O Python lê tudo como UTF-8 e encontra bytes inválidos no meio do caminho.

erro típico
# Linha 1 (UTF-8): "São Paulo" → ok # Linha 47 (cp1252): "Ação" → bytes \xc3\xa7 ≠ \xe7 UnicodeDecodeError: 'utf-8' codec can't decode byte 0xe7 in position 312 # Solução: chardet detecta o encoding import chardet raw = open('dados.csv', 'rb').read() enc = chardet.detect(raw)['encoding'] df = pd.read_csv('dados.csv', encoding=enc)

BOM — Byte Order Mark

O Excel adiciona automaticamente 3 bytes invisíveis (EF BB BF) no início de arquivos UTF-8. Ao fazer df.columns[0], o nome da coluna aparece como "\ufeffNome" em vez de "Nome" — quebrando qualquer join ou seleção por nome.

bom_fix.py
# ❌ sem tratamento df['Nome'] # KeyError: 'Nome' df['\ufeffNome'] # isso que existe # ✅ utf-8-sig remove o BOM automaticamente df = pd.read_csv(path, encoding='utf-8-sig') df['Nome'] # ✓ funciona
unicode + acentos

Normalização de Acentos e Unicode

Por que o mesmo caractere pode ter dois bytes diferentes?

Unicode permite representar o ã de duas formas: como um único codepoint composto (NFC) ou como a + til separados (NFD). Para o olho humano são idênticos; para o computador, são strings diferentes — o que quebra comparações e joins.

unicode_normalize.py
import unicodedata # Normaliza para NFC (forma canônica composta) def normalizar_unicode(v): return unicodedata.normalize('NFC', v) # Remove acentos (para slugs / chaves de join) def sem_acento(v): norm = unicodedata.normalize('NFD', v) return ''.join( c for c in norm if unicodedata.category(c) != 'Mn' ) # "ação" → "acao" sem_acento("ação") # → 'acao'
✔ qualidade

Validação e Qualidade de Dados

Limpeza sem validação é incompleta — não há garantia de que os dados resultantes seguem as regras esperadas. A validação formaliza essas expectativas como contratos verificáveis.

📐

Schema Validation

Ferramentas como Pandera e Great Expectations permitem descrever o formato esperado de cada coluna — tipo, range, padrão regex, unicidade — e verificar automaticamente se o DataFrame está em conformidade.

📊

Data Profiling

O ydata-profiling gera um relatório HTML com distribuições de valores, taxa de nulos, valores únicos, correlações e alertas automáticos de anomalias — sem escrever uma linha de análise manual.

🔁

Testes de Regressão

O pipeline de limpeza pode ser testado como qualquer código de produção. Fixtures com dados problemáticos reais garantem que correções anteriores não sejam revertidas em futuras alterações.

🗂

Auditoria de Mudanças

Cada transformação aplicada ao dado pode ser registrada com antes/depois, data e regra aplicada — essencial para rastreabilidade, compliance e depuração de erros futuros.

★ boas práticas

Boas Práticas de Data Cleansing

  • Dado bruto preservado — uma camada raw imutável garante que as transformações sejam sempre reversíveis e auditáveis (lake pattern).
  • Documentação das regras — cada decisão de limpeza registra quem a definiu, por quê e qual era o problema original que a motivou.
  • Versionamento do pipeline — scripts de limpeza têm o mesmo ciclo de vida que código de aplicação: Git, revisão de código, histórico de mudanças.
  • Validação em produção — assertions e checks em tempo de execução detectam quando novos dados fogem dos padrões esperados (data drift).
  • Colaboração com o domínio — o que é "dado sujo" depende das regras de negócio, não apenas de critérios técnicos.
  • Profiling periódico — relatórios automáticos de qualidade como parte recorrente do pipeline, não só em fase de desenvolvimento.
